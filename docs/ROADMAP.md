# 路线图

本文档收敛此前分散的多份设计草案。以下主题都值得继续推进，但截至 2026-03-25，它们不应被视为仓库默认能力。

## 核心愿景

本 monorepo 以**混合契约架构**为核心，未来计划向 **AI 自举迭代能力**方向发展，同时兼顾人工介入开发，实现**奇点式开发平台**。

### 混合契约架构

- 契约定义是变更入口，驱动代码生成
- 支持多协议（gRPC、HTTP、Connect-RPC）从同一契约生成
- 前后端类型安全，契约变更立即反映到所有端

### AI 自举迭代

- AI Agent 可基于契约自动生成/修改代码
- 人工介入点明确，可随时接管
- 平台自身可被 AI 理解和修改，实现自我进化

## 1. 契约架构升级

目标：

- 实现全栈类型安全，后端修改字段前端立刻报错
- 支持多端客户端（Web、App、小程序、鸿蒙）的契约至上
- 统一 gRPC、Connect-RPC、HTTP 三套接口的权限与流式通信

现状：

- Kratos 后端已提供 gRPC + HTTP 两套接口
- 已自动生成 OpenAPI 规范到 `gen/oas/`
- Connect-RPC 尚未引入
- 前端尚未建立统一的契约消费层

详细决策记录见 [API_CONTRACT_DECISIONS.md](./API_CONTRACT_DECISIONS.md)

## 2. 错误治理

目标：

- 统一错误分类、可观测性、外部暴露格式
- 明确业务错误、基础设施错误、依赖错误的边界

现状：

- 当前已存在 Ent 错误到 Kratos 错误的基础映射
- 还没有跨仓库统一的错误模型规范文档与实现抽象

## 2. i18n

目标：

- 服务端错误消息与响应消息具备多语言能力
- 支持本地兜底与远程翻译源

现状：

- 仓库里还没有 i18n 管理器、语言资源结构、切换策略的实现

## 3. 限流、熔断、降级

目标：

- 在 server / client 侧统一沉淀弹性治理能力
- 为内部调用和对外接口提供不同保护策略

现状：

- 当前中间件链路里尚未看到仓库级默认限流与熔断配置

## 4. 分布式事务

目标：

- 为长事务编排与跨服务一致性提供规范化接入路径

现状：

- 当前只有单服务数据库事务抽象 `biz.Transaction`
- Temporal、DTM 等方案尚未进入代码基线

## 5. 链路上下文增强

目标：

- 让客户端更容易拿到 trace context，并贯穿前后端问题排查

现状：

- tracing 已接入服务端与 SQL
- 当前响应编码器没有注入 `X-Trace-Id` / `traceparent` 的实现

## 6. 推进原则

- 先做最小可用落地，再补全设计稿
- 方案文档必须标注“已实现 / 未实现 / 风险 / 前置条件”
- 新增一类平台能力时，优先考虑是否能沉淀到 `shared-go` 或 `infra`
