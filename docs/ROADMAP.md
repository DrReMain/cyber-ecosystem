# 路线图（非现状承诺）

更新时间：2026-03-26。  
本文档仅描述后续演进方向，不代表默认已实现能力。

## 1. 状态分层

### 1.1 已实现

- gRPC + HTTP + Connect 三协议服务端链路
- 错误语义统一（`reason/message/details`）
- i18n 抽象与默认 go-i18n provider
- `Accept-Language` 权重解析
- 关键元信息 header 注入（trace/success/reason）

### 1.2 进行中

- 生成器与示例持续对齐
- 文档体系去滞后化（规范文档与规划文档分离）

### 1.3 未开始或未形成默认能力

- 客户端统一 SDK（connect-web / ts-rest）
- 限流、熔断、降级统一治理
- 分布式事务编排能力
- 跨网关链路上下文标准化（如 traceparent 全链路规范）

## 2. 重点主题

### 2.1 契约架构升级

目标：

- 完成客户端契约消费统一层
- 明确 Connect 与 HTTP 兼容层边界

下一步：

- 在仓库内落地最小 TS SDK 适配层

### 2.2 错误治理深化

目标：

- 收敛框架级错误与业务级错误对外语义
- 建立跨协议一致性回归测试

下一步：

- 补齐 shared-go 层错误回归测试矩阵

### 2.3 i18n 工具链完善

目标：

- 形成 reason 枚举到 locale 字段的持续同步机制
- 建立缺失字段检测

下一步：

- 将 i18n sync 工具纳入标准开发流程

### 2.4 平台治理能力

目标：

- 引入统一限流/熔断/降级策略
- 明确服务内与服务间治理边界

### 2.5 分布式事务

目标：

- 在 `biz.Transaction` 之外提供跨服务事务方案

## 3. 推进原则

- 先最小可用，再抽象复用
- 每次重构必须同时更新生成器和文档
- 对“已实现”声明必须可由代码或 target 直接验证
