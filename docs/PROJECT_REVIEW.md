# 项目审视（阶段性）

评审时间：2026-03-26。  
范围：仓库结构、生成器一致性、返回与错误体系、文档可信度。

## 1. 结论

仓库已经从“概念验证”进入“可持续演进”阶段，但仍有两个主要风险：

- 生成器与手工演进代码容易再次漂移
- 客户端 SDK 侧协议归一化尚未补齐

## 2. 已改善项

- 返回结构已切换到“成功原生 body + 错误统一语义”
- `common.Reply` 旧模型已退出主路径
- i18n 已具备抽象接口与默认实现，且支持 `Accept-Language` 权重解析
- Connect transport 已进入实际示例服务链路
- 生成器已同步 error reason 与 i18n 基础模板，并通过生成+启动冒烟

## 3. 当前主要问题

1. 客户端消费层缺失：
   目前仅有后端统一语义，前端还没有统一 SDK。
2. 规范执行风险：
   后续如果不持续做“模板冒烟 + 文档同步”，容易回归到历史问题。
3. 文档漂移风险：
   决策文档和路线图如果不标注“已实现/未实现”，会再次误导执行。

## 4. 建议优先级

1. 建立模板改动 CI 校验（至少生成并编译一次）
2. 建立客户端 SDK 归一化 PoC（connect-web + HTTP）
3. 把“文档更新”纳入重构完成定义（DoD）
