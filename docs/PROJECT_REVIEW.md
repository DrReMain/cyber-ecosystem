# 项目审视

本评审基于 2026-03-22 仓库现状，对结构、扩展性、健壮性、稳定性、文档一致性做一次快速审视。

## 1. 主要结论

仓库已经具备可复用的后端基础骨架，但离“稳定的平台型单仓”还有明显距离。当前最需要修复的不是功能缺口，而是“认知层一致性”：文档、生成器、目录分层、命令入口之间存在多处偏差。

## 2. 结构层面

优点：

- `contracts`、`examples`、`shared-go`、`infra`、`tools` 的职责基本分开
- `infra/project.json` 让工具链与 Docker 能力有了统一入口
- `tools/generators/kratos-base` 说明仓库正在从“手工复制模板”转向“脚手架生成”

隐患：

- 生成器默认目录是 `services/`，但现有样例都在 `examples/`
- `gen/` 是产物目录而不是能力入口，名字容易让人误判
- `clients/admin` 目前是占位项目，容易让人误以为前端规范已成型

## 3. 扩展性

优点：

- Proto 契约中心化，适合继续生成 SDK、OpenAPI、跨语言接口
- `shared-go` 已抽出编码器、校验、分页、事务辅助等公共能力

隐患：

- 远程依赖直接挂在 Data 层，项目扩大后可能把 Data 演变成“所有基础设施依赖的收纳箱”
- 事务、错误、观测虽然已有基础，但还没沉淀成统一治理模块
- 生成器还没有完全与当前文档和目录规范对齐

## 4. 健壮性与稳定性

优点：

- HTTP/gRPC 服务已接入 recovery、metrics、tracing、logging
- SQL tracing 已通过 `otelsql` 集成
- Ent 错误有统一映射入口

隐患：

- `NewData` 中连接数据库失败直接 `log.Fatalf`，会让构造函数承担进程退出行为，降低可测试性
- `template2` 的远程 gRPC 连接在初始化失败时 `panic`，错误恢复路径较硬
- 文档中提到的 trace header 注入并未在当前 encoder 中实现，说明“观测能力说明”与代码曾经脱节

## 5. 文档层面

本次整理前存在的问题：

- `README.md` 不是项目入口，而是一段临时错误摘录
- `AGENTS.md` 与 `DEVELOPMENT.md` 大量重复
- 多份计划文档把未实现能力写得像现状
- 部分命令写错了项目名，例如把 `infra:*` 写成 `tools:*`

## 6. 优先建议

1. 继续保持“规范文档”和“路线图文档”分离
2. 统一新服务生成路径策略：是迁移到 `services/`，还是让生成器贴合 `examples/`
3. 把初始化阶段的 `panic` / `Fatalf` 逐步收敛为 error 返回
4. 给 `shared-go` 和两个模板补更明确的测试边界
5. 当某项能力未实现时，文档只写“计划”和“假设”，不要写成“现有行为”
