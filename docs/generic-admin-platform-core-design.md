# 通用后台管理脚手架核心设计

## 1. 文档目的

本文档只描述通用后台管理脚手架的核心设计，不讨论具体语言、框架、目录模板、数据库表结构或代码实现细节。

目标是先回答 3 个问题：

1. 这个脚手架应该提供哪些平台能力。
2. 这些能力应该如何被业务服务接入。
3. 怎样在保持能力完整的同时，避免系统重新演变成一个高耦合的大杂烩。

## 2. 设计目标

该脚手架需要提供以下基础能力，并允许业务服务按需接入：

- Session
- RBAC
- ABAC
- DataScope
- Audit
- Storage

同时满足以下要求：

- 能力是可选接入的，而不是强制全量启用。
- 业务服务可以保留自己的领域模型和部署边界。
- 不把所有判断都集中到一个远程服务上，避免中心服务变成所有请求的硬依赖。
- 对外暴露的是稳定能力契约，而不是业务代码之间的直接耦合。

## 3. 非目标

以下内容不属于本阶段核心设计范围：

- 不定义具体 ORM、消息队列、缓存产品或网关实现。
- 不定义具体 proto、数据库 schema、代码生成模板。
- 不追求“一次设计覆盖所有端和所有业务场景”。
- 不把脚手架本身设计成一个默认承载所有业务的单体系统。

## 4. 核心思路

### 4.1 能力按平台域收敛

不要把 `Session`、`RBAC`、`ABAC`、`DataScope`、`Audit`、`Storage` 拆成 6 个完全平级的独立系统。

更稳定的划分是 3 个平台能力域：

| 能力域 | 包含能力 | 作用 |
|---|---|---|
| `access` | Session, RBAC, ABAC, DataScope | 身份、授权、条件判断、数据可见范围 |
| `audit` | Audit | 安全审计和业务审计 |
| `storage` | Storage | 文件对象与文件元数据管理 |

其中 `RBAC`、`ABAC`、`DataScope` 本质上都属于授权决策系统，不应拆成多套彼此独立的远程能力。

### 4.2 每个能力都分为控制面和运行面

每个能力域都区分为两部分：

- **控制面（Control Plane）**  
  负责配置、规则管理、元数据维护、管理 API，是 source of truth。

- **运行面（Runtime Plane）**  
  负责在业务服务请求链路中执行能力逻辑，例如 token 校验、权限判定、数据过滤注入、审计采集、文件访问适配。

核心原则：

- 规则和配置在控制面管理。
- 请求时的执行尽量在业务服务本地完成。
- 控制面负责下发规则、版本和元数据，不负责承接每个业务请求的实时判定。

这条原则尤其适用于 `access`。

## 5. Access 能力设计

`access` 是整个脚手架最核心的平台能力，统一负责：

- 用户身份确认
- 会话管理
- 权限判断
- 条件判断
- 数据访问范围输出

### 5.1 为什么要统一为一个能力域

如果把 Session、RBAC、ABAC、DataScope 分散设计，最终常见问题是：

- 认证是一套上下文
- 权限判断是一套资源命名
- 条件判断再读另一套上下文
- 数据权限又变成独立规则系统

结果是同一个请求要经过多套不统一的判断链，配置、缓存、调试和审计都会非常混乱。

因此 `access` 应统一输出一份授权决策结果，而不是让多个子系统彼此独立地做结论。

### 5.2 Access 的输入与输出

对任意一个业务请求，`access` 的运行时决策输入至少包括：

- `principal`：当前用户或调用主体
- `resource`：访问的资源
- `action`：执行的动作
- `environment`：时间、IP、设备、来源等环境上下文
- `service metadata`：服务名、操作名、请求附加属性

输出统一为：

- `allow / deny`
- `reason`
- `filters`：结构化数据过滤条件
- `obligations`：额外执行要求，例如记录安全审计、注入上下文等

其中：

- `RBAC` 提供基础授权
- `ABAC` 提供环境条件判断
- `DataScope` 提供结构化数据过滤条件

### 5.3 Session 设计

Session 的核心目标是低成本确认“你是谁”，而不是让每个请求都远程访问认证服务。

建议机制：

- 登录后签发短期 access token。
- token 可在业务服务本地验签。
- token 中包含 `subject`、`session_id`、`realm`、`policy_version` 等核心信息。
- 会话撤销通过短 TTL、版本失效、撤销列表或缓存同步解决。

核心原则：

- 业务服务优先本地校验 token。
- 不把“每个请求远程调用认证中心验证 session”设计成默认路径。

### 5.4 RBAC / ABAC / DataScope 的统一决策

`RBAC` 负责回答“主体是否具备执行某类动作的基本权限”。  
`ABAC` 负责回答“在当前环境和条件下是否允许”。  
`DataScope` 负责回答“允许访问哪些数据，而不是是否允许访问整个接口”。

三者应该统一进入一个授权决策模型，而不是分散为三套独立中间件体系。

尤其对 `DataScope`，设计原则必须明确：

- 输出结构化过滤条件，不输出拼接后的 SQL。
- 规则属于控制面。
- 过滤注入属于运行面，由业务服务本地的 Repo / ORM 适配层负责落地。

## 6. Audit 能力设计

`audit` 独立成平台能力，负责两类审计：

| 类型 | 内容 |
|---|---|
| 安全审计 | 登录失败、鉴权拒绝、条件拒绝、会话失效、越权访问 |
| 业务审计 | 创建、修改、删除、导出、审批等业务操作 |

### 6.1 Audit 的定位

`audit` 不负责拦截业务逻辑，也不负责权限判断。  
它只负责可靠地接收、存储、查询审计事件。

### 6.2 Audit 的接入方式

业务服务接入 `audit` 时，应通过统一审计客户端或审计 SDK 写入事件，而不是在各服务中随意定义自己的日志结构。

设计原则：

- 审计事件模型统一。
- 失败和拒绝事件可以由运行时自动采集。
- 业务成功事件可以由业务服务显式记录。

### 6.3 Audit 的可靠性要求

审计不是普通 debug log。

因此默认语义应是：

- 可以异步发送。
- 不能静默丢失。
- 至少要能区分“已写入”“待重试”“最终失败”。

无论底层用 outbox、可靠队列还是持久化缓冲，原则都不应变成“发送失败就直接忽略”。

## 7. Storage 能力设计

`storage` 负责文件对象和文件元数据，不直接承载业务领域逻辑。

### 7.1 Storage 的职责边界

它应该负责：

- 文件上传、下载、删除
- 文件元数据维护
- 文件生命周期状态管理
- 对接本地文件系统、MinIO、S3 等物理存储后端

它不应该负责：

- 业务对象审批逻辑
- 业务权限建模
- 各种业务表与文件表之间的复杂联动规则

### 7.2 文件生命周期

建议至少区分以下状态：

- `pending`
- `committed`
- `orphaned`
- `deleted`

这样业务服务可以先获得文件 ID，再决定是否真正把文件与业务实体建立关系，孤儿文件则由平台定期清理。

## 8. 可选接入机制

### 8.1 能力按服务声明接入

每个业务服务都应显式声明自己接入哪些能力，而不是由平台隐式注入。

例如一个业务服务可以声明：

```yaml
session:
  mode: enforce

authorization:
  mode: enforce

audit:
  mode: async

storage:
  mode: disabled
```

### 8.2 模式必须收敛

建议所有横切能力都复用统一模式语义：

- `disabled`：完全不参与
- `observe`：执行但不阻断，只记录结果
- `enforce`：执行并严格阻断

其中 `observe` 非常重要。它允许新能力先在生产链路中验证效果，再切到强制执行。

## 9. 资源与权限的 Source of Truth

权限系统最容易失控的地方，不是算法，而是资源命名漂移。

因此必须有统一资源注册机制。

每个业务服务应声明自己的：

- 资源命名空间
- 资源类型
- 动作列表
- 可用于 DataScope 的字段元数据

控制面以此作为：

- 权限配置来源
- 管理后台展示来源
- 审计分类来源
- 授权运行时的元数据来源

核心原则：

- 资源定义不能散落在业务代码字符串里。
- 权限模型和管理界面不能各自维护一套资源词典。

## 10. 请求链路建议

对一个典型业务请求，推荐的执行链路为：

1. 业务服务接收请求。
2. 本地进行 Session / token 校验。
3. 本地执行统一授权决策。
4. 将 DataScope 过滤条件注入查询层。
5. 执行业务逻辑。
6. 产生审计事件。
7. 如需要，调用 Storage 能力。

这里的关键点不是顺序本身，而是：

- 认证和授权决策尽量本地完成。
- DataScope 不绕过数据访问层。
- Audit 和 Storage 是能力调用，不反向侵入业务模型。

## 11. 依赖关系原则

为了避免脚手架重新演化成新的大单体，必须坚持以下边界：

- 业务服务依赖平台能力契约，不依赖其他业务服务内部实现。
- 平台能力通过 SDK、客户端、中间件、拦截器等稳定接入面暴露能力。
- 控制面服务不直接承载业务领域逻辑。
- 共享模块不应依赖某个具体业务 App 的私有协议定义。

尤其要避免一种错误设计：

> 名义上是“共享能力模块”，实际却反向依赖某个具体产品或某个具体业务 App 的代码与协议。

一旦这样设计，所谓共享能力只是把耦合换了个目录继续存在。

## 12. 演进建议

脚手架不应一开始把所有能力一次做满。

更合理的演进顺序是：

1. `access` 先落地 `Session + RBAC`
2. `audit` 落地统一事件模型和可靠采集
3. `storage` 落地文件元数据与对象存储抽象
4. `access` 再补 `ABAC`
5. `access` 最后补 `DataScope`

原因是：

- `Session + RBAC` 是最基础也最容易形成闭环的能力。
- `Audit` 和 `Storage` 边界相对清晰。
- `DataScope` 最容易把授权系统、数据访问层和业务模型重新耦合在一起，应该最后设计和接入。

## 13. 结论

通用后台管理脚手架的关键，不是预置尽可能多的模块，而是建立一套长期稳定的能力边界。

最核心的结论可以归纳为 4 条：

1. 平台能力应收敛为 `access`、`audit`、`storage` 三个能力域。
2. 每个能力都应区分控制面和运行面，规则集中管理，执行尽量本地完成。
3. 能力接入必须显式声明，并支持 `disabled / observe / enforce` 三种模式。
4. 共享能力必须依赖稳定平台契约，不能反向依赖具体业务产品。

如果这 4 条没有守住，系统即使看起来拆成了多个服务，最终也仍然会重新回到高耦合、难演进、难裁剪的状态。
